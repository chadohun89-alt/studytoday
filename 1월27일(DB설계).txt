<회원>
기본키 값, 이메일, 비밀번호, 닉네임, 생년월일

member 테이블
기본키 값, 이메일, 비밀번호, 닉네임, 생년월일, 이미지경로
랭킹, 업적목록

테이블 설계
	0 정규화란? > 데이터를 잘게 나눠 규칙(질서)를 정하는 작업
		      데이터가 커지면 꼬임, 중복, 오류가 발생되는데 이를 막기위한 방법
		      공식 정의 : 데이터 중복과 이상현상을 제거하기 위해 <테이블을 구조적으로 분해>하는 과정

	1 이상현상

		1-1 삽입 이상 ex) 이창호 등록안 됨. 국문과라는 과가 존재하지 않음. 학과테이블에 국문과가 없어서.
			> 학과명이 없어 학생을 등록 못한다.
			   학원에서 새로운 과정을 개설했다. 새 과정을 테이블에 저장 해야되는데 아직 수강생이 없어 등록 못한다.

		1-2 갱신 이상 (데이터 업데이트 - 변경 시 문제 되는 것)
			> 같은 정보가 여러 행에 중복 돼 있어, 일부만 수정하면 데이터가 달라지는 문제

		1-3 삭제 이상 (삭제에서 문제)
	 		> 어떤 데이터를 지웠더니 원치 않는 다른 데이터까지 같이 사라지는 문제

	2. 정규화 원칙

		2-1 하나의 테이블은 하나의 주제
		2-2 하나의 컬럼에는 하나의 값
		2-3 기본키로 모든 데이터 조회가 가능해야함
		2-4 파생 정보는 저장하지 않는다.
		  -> 생년월일 저장했는데 나이를 또 저장하는 것

	3. 정규화 단계
		
		3-1 제1정규화(1NF)
			> 테이블의 모든 컬럼에는 더이상 쪼갤 수 없는 원자값이어야 한다.
		3-2 제2정규화(2NF)
			> 제1정규화를 만족하고 모든 칼럼이 기본키에 완전 종속 된다.
			# 종속판별
				- A값이 같으면 B값도 항상 같은가?
					ex) A가 같은 행들이 여러개 있을 때 B가 항상 동일 해야 A -> B 성립
					    id, student_id, student_name 컬럼이 있다면
					    student_id -> student_name 성립이 되는지

				- A 하나만으로 B를 결정할 수 있는가?
					ex) A만 알면 B가 딱하나로 확정 되는가?
				- 예외 케이스가 존재 하는가?
					* 종속은 대부분이 아니라 항상이어야 한다.
					* 예외가 1개라도 있으면 그 종속은 성립하지 않거나 설계 규칙이 틀린거다.
					* 데이터 저장이 잘못되었거나 업무규칙이 애매하다. 			
				- 항상 하나로 정해지면 종속!

				- 한 과정(course)에 여러 학생이 수강한다.
				- course_id -> student_id

				- 프로젝트 팀구성에서 한 팀 안에서 팀원은 역할을 1개 갖는다.
				- (team_id, member_name) -> role

				- 과정별 담당 강사가 기간에 따라 변경될 수 있다.
				- course_id -> teacher_name

				- 학생 id는 학원에서 유일하다.
				- student_id -> student_name

				- 과정ID는 과정 하나를 의미한다.
				- course_id -> course_name

				- 과정명이 변경될 수 있고, 중복 이름도 가능하다.
				- course_name -> course_id

				- 학생은 같은 과정에 한 번만 등록 가능하다.
				- (student_id, course_id)  -> enroll_date

		3-3 제3정규화(3NF) (실무에서는 종착역)
		3-4 보이스 - 코드 정규화(BCNF) (기본 키가 아닌 후보 키로 설계하는 것
		3-5 제4정규화(4NF) (자격증 시험에서는 나옴, 실무에서 거의 안 씀)
		3-6 제5정규화(5NF) (논문에서나 쓰는 정규화)
		3-7 도메인 - 키 정규화(DKNF) (이론적으로만 존재함)
		3-8 제6정규화(6NF)	 (막대한 데이터를 전제로함)
